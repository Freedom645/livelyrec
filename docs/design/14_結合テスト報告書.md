## 4. 検出欠陥（I-009〜I-017 / 全件 Closed）

| ID | 等級 | 検出 | 概要 |
|----|------|------|------|
| I-009 | 重大 | 区分A IT-WS-04 | ws_server が websockets 16 の旧 API を参照し認証パスが落ちる |
| I-010 | 致命 | 区分B IT-OBS-01 | 再接続の無限再帰で OBS 接続嵐＋RecursionError |
| I-011 | 重大 | 区分B IT-OBS-01 | ソース名未設定（設定エラー）を接続エラーと誤分類 |
| I-012 | 重大 | 区分B IT-OBS-01 | UI 更新がワーカースレッドから届かない（QTimer 誤用） |
| I-013 | 軽微 | 区分B IT-OBS-01 | connect が識別失敗を検知しない |
| I-014 | 重大 | 区分B IT-BROWSER-01 | WebSocket Server が HTTP 静的配信に未対応でブラウザソースを開けない |
| I-015 | 致命 | 区分B IT-FULL-01 | app.py が楽曲マスタを DB にロードせず、譜面特定不能で記録されない |
| I-016 | 重大 | 区分B IT-OCR-01/FULL-01 | リザルト score/combo が装飾フォントで OCR 不可、clear_type 未検出 |
| I-017 | 重大 | 区分B IT-OCR-01 | リザルト記録が最初のフレームで即時に行われ、アニメーション途中の値を読む |

### 4.1〜4.4

I-009〜I-016 の詳細は本書 v0.2／v0.3 を参照。要点:
- **I-009〜I-013**: OBS 接続層を `simpleobsws`（非同期）→ `obs-websocket-py`（同期）へ移行、`recording_service` を同期ループ＋有界再接続へ再設計。
- **I-014**: `ws_server` に `process_request` で `browser_source` の HTTP 静的配信を追加。
- **I-015**: 起動時に楽曲マスタを DB へロード（endpoint 取得＋同梱 seed フォールバック）。「記録されない」の主因。
- **I-016**: リザルト score/combo を digit テンプレートマッチング方式へ。不足の 7/9 テンプレを生成。clear_type 未検出時は CLEAR 既定。

### 4.5. I-017（リザルト記録タイミング）

- **検出**: IT-OCR-01 再実施。スコアが不正確、判定数が記録されない。ユーザ所見「表示しきる前に判定している」。
- **原因**: pop'n music のリザルト画面はスコア・判定数がカウントアップ表示される。`_handle_result` が最初のリザルトフレームで即記録するため、アニメーション途中の値を読んでいた。
- **対応**: `_ResultStabilizer` を導入。スコアが数フレーム同値で安定（＝アニメーション完了）してから記録する。安定しない場合も一定フレーム後にフォールバック記録。

## 7. 申し送り・観察事項（工程8 システムテストへ）

1. **楽曲名 OCR の認識精度**: 実プレイ画面での OCR・ROI 調整。実プレイ録画をもとに校正する。
2. **画面誤判定の精度**: ロード画面等での画面判別の誤りを実footage で改善する（IT-OBS-02 指摘）。
3. **リザルト clear_type の確実な判定**・**プレイ画面コンボ（クロック型数字）の認識**（I-016 で繰越）。
4. **数字テンプレートの補強**: 7/9 テンプレートは単一サンプル由来（R-022 Watching）。複数サンプルから再生成する。
5. **リザルト記録タイミング（I-017）の実機確認**: `_ResultStabilizer` の安定化フレーム数を実プレイで微調整する。
6. **デバッグ撮影機能**: 記録中フレームを `livelyrec_data/debug/` に保存する機能を活用し、上記の校正を行う。
