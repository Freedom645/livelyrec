## 1. 目的

選曲画面 (SELECT) およびリザルト画面 (RESULT) でのバナー画像を用いた楽曲特定方式の実現性を検証する。

### 1.1. 背景

- 既存方式（PoC #01）ではプレイ画面の楽曲名 OCR が一次ソース。
  - 「ぽぽぽかレトロード」が「ほかぼかレトロード」に誤認識されるなど、清音/濁音/半濁音の取り違えが残存。
  - SongStabilizer による多数決と rapidfuzz による master.json ファジーマッチで補完中。
- リザルト画面の楽曲名バナーは OCR 困難（PoC #01 で 0/6）。現状は直近キャッシュ依存。
- 既存実装 (`livelyrec/infrastructure/banner_writer.py`) で「リザルト画面バナーの ROI 切り出し→PNG 保存」は実装済み (FR-DEV-002)。本 PoC はその「特徴量と楽曲名の対応データ構築」を扱う後段工程。

### 1.2. 解決したい課題

| ID | 課題 | 解決方針 |
|----|------|---------|
| Q-04-1 | リザルト画面で楽曲特定ができないケースが残る | バナー画像特徴量と楽曲対応表で OCR フォールバック |
| Q-04-2 | 選曲画面のカーソル位置の楽曲特定が未実装 | 同上の対応表で SELECT 画面側の楽曲特定にも応用 |

## 2. 評価軸と合格基準

| # | 指標 | 合格基準 | 備考 |
|---|------|---------|------|
| 1 | Wiki サイトから lively 収録曲のバナー回収率（master.json 楽曲との照合一致率） | ≥ 70% | サイト名命名揺れと lively のジャンル名表記により完全 100% は困難 |
| 2 | 特徴量による Top-1 マッチ精度（既存 RESULT サンプル6枚に対する正解率） | ≥ 90% | OCR の補完であり、確信が低ければ既存ロジックに委譲する設計のため緩め |
| 3 | 1 枚あたりの特徴量計算時間 | ≤ 50ms | 認識ループ内で 1FPS 程度の呼び出しを想定 |
| 4 | 1 枚あたりの Top-K マッチング時間（1347 曲想定） | ≤ 30ms | pHash 64bit のハミング距離ソートを想定 |

## 3. 検証スコープ

### 3.1. 対象サイト

| サイト | 評価項目 | 想定リスク |
|--------|---------|-----------|
| remywiki.com (`Category:Pop'n_music_Banners`) | MediaWiki API でカテゴリ列挙可能。ファイル命名 `BN_<曲名>.png` 系 | pop'n 全シリーズ網羅のため lively 非収録曲も多数 |
| popnmusic.fandom.com (`Category:Song_Banners`) | 同上の MediaWiki ベース。Fandom 規約で UA 必須 | 同上 |

### 3.2. 採用候補の特徴量

| 方式 | 計算量 | UI 装飾耐性 | 想定用途 |
|------|--------|----------|----------|
| pHash (64bit) | 低 | 中（縁取りに弱い） | 1 次絞り込み |
| dHash (64bit) | 低 | 中 | 1 次絞り込み代替 |
| ORB 特徴点記述子 | 中 | 高 | 2 次確認 |
| CNN 埋め込み (ResNet 等) | 高 | 高 | 採用しない（PoC #01 で重量級依存は懸念済み） |

**採用方針候補**: pHash で Top-K 絞り込み (K=5) → ORB マッチで Top-1 確定 のハイブリッド。

### 3.3. 配布方針

KONAMI 著作物（バナー画像本体）の二次配布を避けるため:
- アプリ同梱は `data/banner_features.json`（song_id, pHash, ORB 記述子要約のみ）
- 画像本体は **ユーザ PC のローカルキャッシュ** (`livelyrec_data/banners_ref/`) として運用
- マスタ生成スクリプトは初回実行時にユーザ PC でダウンロードする（同意プロンプト想定）

## 4. 実施手順

### 4.1. 収集スクリプト (`scripts/poc/banner_collect.py`)

1. MediaWiki API でカテゴリページからファイル名一覧を取得（remywiki / fandom 両対応）
2. 画像 URL を取得（`prop=imageinfo`）
3. レート制御（1 リクエスト毎 1 秒以上）で画像 DL
4. ローカル `livelyrec_data/banners_ref/<site>/<filename>` に保存
5. UA は `LivelyRec-PoC/0.1 (https://github.com/OWNER/lively-)` 明示

### 4.2. マッチング試作 (`scripts/poc/banner_match.py`)

1. ローカル DL 済みの画像ファイル名 → master.json タイトルへの rapidfuzz マッチ
2. ファイル名前処理: `BN_`/`_banner` 等の接頭/接尾辞除去、`_` を空白へ
3. lively ジャンル名対応（既知ジャンル名リストを別ファイルで用意可能な構造）
4. 一致した画像セットに対し pHash・ORB を計算
5. `data/banner_features.json` を出力（song_id, pHash, ORB descriptor sample）

### 4.3. 認識精度測定

1. 既存サンプル `tests/fixtures/sample/リザルト画面/`（6 枚）から `RESULT_ROI["banner"]` (489,233,879,327) を切り出し
2. 上記特徴量マスタとマッチング → 正解率を集計
3. SELECT 画面サンプルからも `READY_ROI["song_banner"]` (560,130,980,220) で切り出し試行

### 7.2. サイト API 疎通

| サイト | API | 結果 |
|--------|-----|------|

### 7.3. 試験 DL 結果（remywiki 先頭 30 件）

- バナー画像サイズ: **246×60 px**（pop'n music CS1〜10 のジャンルバナー）
- これは lively の規格 **390×94 px** とは異なる。`cv2.resize` で正規化する処理は機能
- ファイル命名: `<シリーズ番号>_<ジャンル名>.png`（例: `1_DISCO_KING.png`、`10_CYBER_ARABIAN.png`）

### 7.4. lively 楽曲との突合（タイトルマッチ精度）

`rapidfuzz WRatio` でファイル名（ジャンル名）↔ `master.json` 楽曲タイトルを照合。

| 指標 | 結果 |
|------|------|
| ファイル数 | 30 |
| score_cutoff=60 で何らかのヒット | 25 / 30 (83.3%) |
| **真の正解一致** | **0 / 30 (0%)** |

**理由**: 試験 DL は CS1〜10 のジャンルバナーで、**lively 収録曲ではない**ため正解がそもそも master.json に存在しない。score=60〜72 は全件偽陽性。

→ 合格基準 1（回収率 ≥ 70%）は **「全カテゴリ無差別 DL」方式では達成不能**と判明。

### 7.5. 特徴量計算性能（合格基準 3・4 検証）

参照集合 30 件 + 1347 件相当の Top-K に拡張ベンチ。

| 指標 | 結果 | 合格基準 | 判定 |
|------|------|---------|------|
| pHash64 計算 / 枚 | **0.089 ms** | ≤ 50 ms | ✅ 約 560 倍の余裕 |
| dHash64 計算 / 枚 | 0.083 ms | — | ✅ |
| Top-5 マッチング（1347 件、Hamming sort） | **0.357 ms** | ≤ 30 ms | ✅ 約 84 倍の余裕 |

→ 認識ループ内で 1 枚あたり計算+マッチが **1ms 未満**で完結する。PoC #01 の PaddleOCR が 1 枚 463 ms だったのと比べると圧倒的に軽量。

### 7.6. 認識動作確認（合格基準 2 の予備測定）

既存のリザルト画面サンプル 6 枚から `RESULT_ROI["banner"]` (489,233,879,327) を切り出し、参照集合 30 件と pHash + dHash の合計ハミング距離で Top-5 を取得。

参照集合に正解が含まれていない条件下での結果:

| サンプル | Top-1 ハミング距離 (pHash) | 備考 |
|---------|--------------------------|------|
| 12-20-06.png | 24 | 同一曲群（A 曲）で同一 Top-1 |
| 12-22-44.png | 24 | 同一 |
| 12-23-54.png | 24 | 同一 |
| 12-27-48.png | 22 | B 曲 |
| 12-27-52.png | 24 | C 曲群で同一 Top-1 |
| 12-28-30.png | 24 | 同一 |

判定:
- **正解が含まれない条件での Top-1 距離は 22〜30 ビット**（64bit の約 1/3〜1/2 が反転）。**しきい値設計の上限**として有用。
- **同一曲のサンプル群は同一 Top-1 に収束**（決定論的に動作）。リサイズ・ROI 切り出しは安定。
- 正解を含む参照集合での **真の Top-1 距離（同一画像なら 0、UI 装飾差で +5〜10 想定）** は本 PoC では測定不可（lively バナー実画像の取得が必要）。

→ 合格基準 2（Top-1 ≥ 90%）は本 PoC では未測定。本実装フェーズで lively 楽曲ページから真のバナーを集めた上で再評価する。

## 8. 本実装への申し送り

### 8.1. バナー収集ルート（最重要）

「`Category:Pop'n_music_Banners` 全件 DL → master.json 突合」は **lively 収録曲の絞り込みができず非効率**。
代替ルート:

1. **方式 R1（推奨）**: `remywiki.com/wiki/CS_pnm_Lively` ページから lively 収録曲リンクを抽出 → 各曲ページに貼られたバナー画像のみを DL
2. **方式 R2**: master.json の各曲名 / ジャンル名で MediaWiki `list=search` を叩き、対応ページを引き当て → そのページから画像 URL を抽出
3. **方式 R3**: fandom も同様の経路で補完

→ 本実装で `scripts/build_banner_master.py` を作る際は R1 を主、R2/R3 を補としたフォールバック戦略を採用する。

### 8.2. タイトルマッチング設計

- lively は「ジャンル名」と「楽曲名」の二重表記仕様であり、`master.json.title` はジャンル名であるケースが多数。
- `master.json` に **`genre` / `alt_titles[]` フィールドを追加**し、ジャンル名 ↔ 楽曲名のマッピングを保持する設計が必要（A-2 残課題と統合）。
- `rapidfuzz` の score_cutoff は **85 以上**を推奨（PoC では cutoff=60 で全件偽陽性）。

### 8.3. 特徴量データ構造（アプリ同梱形式）

`data/banner_features.json` のスキーマ（案）:

```json
{
  "version": "<iso8601>",
  "target_size": [390, 94],
  "songs": [
    {"song_id": "popn-xxxxxxx", "phash": 1234567890123456789, "dhash": 9876543210987654321, "src": "remywiki:File:..."}
  ]
}
```

- アプリ同梱は **ハッシュ値 + song_id + 出典のみ**。画像本体は同梱しない（§3.3）。
- pHash と dHash の **両ハッシュの合計ハミング距離**を 1 次スコアにする方式が PoC で良好に振る舞った。

### 8.4. 認識しきい値設計

PoC の「ノイズフロア（正解なし時の Top-1 距離）」は **22〜30 ビット**。本実装での想定:

| 状況 | pHash+dHash 合計距離（128bit 中） | 判定 |
|------|----------------------------------|------|
| 一致（同一バナー）| 0〜20 程度 | 高確度 ✅ |
| 部分一致（アレンジ違い等） | 20〜40 | 中確度（OCR 結果で補完） |
| 不一致（無関係） | 40〜 | 低確度（既存ロジックに委譲）|

具体数値は本実装フェーズの正解付き測定で再キャリブレーション。

### 8.5. 性能予算

- リザルト画面では 1 枚 / 数秒オーダーの呼び出しで十分。pHash 0.09 ms + Top-5 over 1347 = 0.36 ms なので予算に対し過剰に高速。
- SELECT 画面でのカーソル位置認識でも 1 FPS 〜 数 FPS でループしてよい余裕。

### 8.6. 法的・運用面

- KONAMI 著作物の二次配布回避のため、**画像本体はユーザ PC ローカル保存**で運用方針を維持。
- 収集スクリプト初回起動時に **同意プロンプト**を出し、UA とリクエスト間隔（≥ 1 秒）を遵守。
- Wiki 側 ToS / robots.txt に追従できるよう、収集スクリプトに `--rate-sec` パラメータを残す。

## 10. 追加調査（2026-05-29、v0.2）

PO からの 3 点の追加観点について調査結果をまとめる。

### 10.1. CS_pnm_Lively ページと master.json の過不足

#### 10.1.1. 調査スクリプト

`scripts/poc/analyze_lively_coverage.py` を新規作成。remywiki の `CS_pnm_Lively` ページの全リンクを `prop=links` で取得し、シリーズページ等のメタリンクを除外、残った候補を `master.json` と rapidfuzz (cutoff=85) で突合。

#### 10.1.2. 結果（生）

| 項目 | 値 |
|------|---|
| CS_pnm_Lively ページの総リンク数 | 795 |
| 楽曲候補（メタリンク除外後） | 731 |
| master.json の楽曲数 | **1347** |
| ファジーマッチ成立件数（cutoff=85） | 340 |
| Wiki 側未マッチ | 391 |
| master.json 側未マッチ | 1055 |

#### 10.1.3. 結果（解釈）

- **Wiki 側網羅が圧倒的に不足**: 731 候補 vs 公式 1347 曲。Wiki は lively の最新追加楽曲を追っていない見込み。
- **「マッチ」340 件の大半は偽陽性**: remywiki のページ名は **ローマ字読み**（例: `Nishi-Shinjuku seisou kyoku`）であり、master.json の公式表記（漢字・かな・英字）と直接照合できない。
  - 偽陽性例: `1 cool no otoko` → `le mal du pays,numéro 1`（共通する '1' で score=85 に乗ってしまう）
  - 真陽性例: `100sec. Kitchen Battle!!` → `100sec. Kitchen Battle!!`（英字楽曲名のみ正確）

#### 10.1.4. 結論と本実装方針

- **「master.json を全量網羅」の原則は維持**: PO 方針通り、公式サイトを基準とする。Wiki は補助情報源。
- **Wiki カバー漏れ分は自前収集（BannerWriter）で埋める**: 既存実装 (`livelyrec/infrastructure/banner_writer.py`) の出力を時間をかけて蓄積。
- **正確なタイトル突合**: 個別楽曲ページの Song Information テーブルから **原タイトル（Japanese title）**を抽出する `prop=parsetree` 等の追加実装が本実装で必要。
- **マッチング戦略の3層化**:
  1. 公式表記 + 原タイトル直マッチ (cutoff=95)
  2. 正規化 + ジャンル名マッチ (cutoff=85)
  3. 上記いずれも未マッチの楽曲は「特徴量未登録」として運用（OCR のみで認識）

→ Wiki 起点でのカバー率は楽観値で **30〜40%** と見積もる。残り 60〜70% は BannerWriter による自前収集を主とする運用設計が必要。

### 10.2. SQLite への登録・更新の考慮

#### 10.2.1. 既存実装

- `livelyrec/infrastructure/repository/schema.py` に `song` テーブル定義あり
  - カラム: `song_id, title, title_norm, genre, has_upper, fetched_at`
  - インデックス: `idx_song_title_norm`
- `livelyrec/infrastructure/repository/song_repo.py` の `SongRepository.upsert()` は **ON CONFLICT DO UPDATE** で更新対応済み
- `livelyrec/application/master_service.py` の `MasterService` が `refresh()`（GitHub Releases 経由）と `load_from_file()`（ローカル JSON）の両方で master.json → DB upsert を行う

#### 10.2.2. バナー特徴量データの格納方式（3案）

| 案 | 形式 | メリット | デメリット |
|----|------|---------|-----------|
| A | `song` テーブルに `phash` `dhash` `banner_src` 列追加 | クエリ単純、JOIN 不要 | 1 曲複数バナー（カバー違い）に非対応、schema_version 昇格必要 |
| B | 新規 `song_banner` テーブル | 1 曲複数バナー対応、出典追跡可 | テーブル追加・マイグレーション必要、JOIN コスト |
| C | DB に入れず `data/banner_features.json` を独立配布 | 既存 master.json と同パターン、schema 変更不要 | 起動時メモリロード必要（ただし 1347 件 × 16bytes = 22KB 程度で問題なし） |

#### 10.2.3. 推奨案

**案 C（独立 JSON ファイル）**。理由:

1. PoC 結果で全件メモリ上の Top-K マッチが **0.36ms / 1347件** と桁違いに速いため、SQLite 化による検索性能向上の恩恵が無い。
2. 既存 `MasterFetcher` の ETag/差分配信パターンをそのまま流用できる（GitHub Releases から `master.json` と `banner_features.json` を並行 fetch）。
3. master.json の更新と独立に banner_features.json を更新できる（Wiki 巡回頻度と公式更新頻度が異なる）。
4. **schema_version 昇格不要**で既存 DB 破壊リスクなし。

#### 10.2.4. 設計案（案 C 採用時）

- 配置: `data/banner_features.json`（リポジトリ同梱）／GitHub Releases へホスト
- アプリ起動時にメモリへロード（`BannerMatchService` 新設、layer は application 層）
- 認識時は `song_id` で `song_repo.get()` と組み合わせて使用
- `banner_features.json` のバージョンと `song` テーブルの `fetched_at` が乖離した場合、UI に警告バーを表示（既存の master 警告バーと同じ実装パターン）

#### 10.2.5. 案 A/B が望ましいケース

- 「全データを SQLite 一元化したい」「DB バックアップだけで完全に復元したい」運用要件があれば案 B が筋。
- 案 A は 1 曲複数バナー（カバー版・lively 限定アレンジ等）への対応が後で困難になるため非推奨。

→ **PO 判断ポイント**: 「SQLite 一元化を強く推す」か「JSON 独立配布で OK」かを基本設計フェーズで決定。

### 10.3. 著作権・法律の観点

**重要**: 以下は技術的整理であり、最終的な法的判断は PO・法務確認が必要。

#### 10.3.1. 関係する権利・契約

| 対象 | 権利 / ライセンス |
|------|------------------|
| バナー画像本体 | KONAMI 著作物 |
| 楽曲タイトル | 著作権保護対象外（事実情報） |
| remywiki 記事テキスト | ライセンス不明（明示なし） |
| popnmusic.fandom テキスト | CC BY-SA 3.0（属性表示・継承義務） |
| Wiki 掲載のバナー画像 | KONAMI 著作物のままで、Wiki は fair use 主張で掲載 |

#### 10.3.2. リスク評価

| アクション | リスク | 緩和策 |
|-----------|--------|-------|
| Wiki から画像をダウンロード（ユーザ PC ローカル） | **低**（私的複製の範囲。日本著作権法 30 条） | ローカルキャッシュ用途に限定 |
| ハッシュ値（pHash/dHash 64bit）の計算と保存 | **低**（元画像復元不能の一方向ハッシュ。複製とは性質が異なる） | ただし「複製」性は判例未確立。安全側で出典明記 |
| ハッシュ値のアプリ同梱配布 | **低〜中**（数値情報の二次配布。技術的には複製とみなされにくい） | §10.3.4 の対策を併用 |
| **画像本体のアプリ同梱配布** | **高**（KONAMI 著作物の二次配布） | **行わない** |
| Wiki テキストの転載（曲リスト等） | 中（fandom は CC BY-SA 義務、remywiki は不明） | 必要最小限・属性表示 |
| Wiki への高頻度クロール | 中（サーバ負荷・ToS 抵触） | レート制御 ≥ 1.0 秒/req、UA 明示 |

#### 10.3.3. 安全運用の設計原則（PoC §3.3 を強化）

1. **画像本体は決してアプリ同梱しない**。常にユーザ PC ローカル `livelyrec_data/banners_ref/` に保管。
2. **アプリ同梱は数値ハッシュ + song_id + 出典（"remywiki:File:..."）のみ**。
3. **初回起動時に同意プロンプト**: 「ユーザの責任において、参照 Wiki から個人利用範囲でバナー画像を取得します。設定で無効化可能」を表示。
4. **Wiki への配慮**:
   - User-Agent に連絡先 URL を明示
   - レート制御 1.0 秒以上
   - 取得済みファイルは再 DL しない（ローカルキャッシュ優先）
   - robots.txt 遵守
5. **属性表示**: 設定画面の「About」欄に出典を明記（"バナー画像参照: remywiki.com / popnmusic.fandom.com"）。
6. **オプトアウト**: ユーザ設定で「Wiki からの自動収集を無効化」可能。無効時は BannerWriter による自前収集のみ。

#### 10.3.4. 推奨案: 「ハッシュ値のみ配布」+「画像はユーザ DL」

- アプリ同梱: `data/banner_features.json`（数値ハッシュ + song_id + 出典）
- ユーザ操作: 設定画面で「Wiki からバナー画像取得」を実行 → ローカル `livelyrec_data/banners_ref/` に保存
- アプリ動作: ハッシュ値で楽曲特定→ローカル画像があれば配信支援 UI に表示、なければ楽曲名のみ表示

この方式なら **KONAMI 著作物の二次配布を完全に回避**できる。

#### 10.3.5. 残リスクと PO 判断事項

- 「ハッシュ値の二次配布」が「複製の頒布」に該当するかは日本判例未確立。**PO・法務に最終判断を仰ぐ必要あり**。
- Wiki への自動アクセス自体（クロール）は技術的には標準的だが、各 Wiki の ToS を改めて確認する必要あり。
- KONAMI が「ファンメイドの認識アプリ」をどう扱うかは個別判断。トラブル回避のため、配布時に「本アプリは KONAMI 公式とは無関係です」と免責表示を推奨。

## 11. PO 判断（2026-05-29 確定）

§10 の追加調査を受け、PO により以下の方針が確定した。本実装フェーズ以降の前提とする。

### 11.1. データ格納・配布方式: 案 C（JSON 独立配布）

§10.2.3 推奨の **案 C** を採用する。

- 配布物: `data/banner_features.json`
- 配信経路: GitHub Releases（既存 `master.json` と並行ホスト）
- アプリ動作: 起動時メモリロード（1347 件想定で約 22KB）
- SQLite スキーマは **変更しない**（schema_version 据え置き）
- `MasterFetcher` と同パターンで ETag/差分取得を実装

### 11.2. ハッシュ値同梱配布: 採用可

ハッシュ値（pHash/dHash 64bit）のアプリ同梱配布を **採用可** と判断。

根拠:
1. 本アプリは **非営利目的**（個人・コミュニティ向けスコア記録ツール）である
2. 同梱データは **著作物を直接扱わない**（一方向ハッシュの数値のみで、元画像復元は不可能）
3. KONAMI 著作物（バナー画像本体）はユーザ PC ローカル保存のみで、二次配布は行わない

→ §10.3.4「ハッシュ値のみ配布」+「画像はユーザ DL」方式で確定。

### 11.3. 安全運用は §10.3.3 の原則を引き継ぐ

PO 判断にあたり、§10.3.3 の以下原則は本実装でも遵守する:

1. 画像本体は決してアプリ同梱しない
2. アプリ同梱は数値ハッシュ + song_id + 出典のみ
3. 初回起動時の同意プロンプト + 設定でのオプトアウト
4. Wiki アクセス時の UA 明示・レート制御 ≥ 1.0 秒・robots.txt 遵守
5. About 欄に出典属性表示
6. 「KONAMI 公式とは無関係」の免責表示
