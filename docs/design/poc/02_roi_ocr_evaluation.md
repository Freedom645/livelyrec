## 1. 目的

PoC #01 で全画像OCRは合格基準（≤200ms/枚）に届かなかった一方で、領域限定OCR＋マスタファジーマッチングで実用精度・速度が見込めるとの仮説を立てた。
本PoC #02 でこの仮説を検証する。

## 2. 評価軸と合格基準

| # | 指標 | 合格基準 |
|---|------|---------|
| 1 | プレイ画面の楽曲名識別精度（マスタファジーマッチ後） | ≥ 95% |
| 2 | リザルト画面のスコア・コンボの取得 | ほぼ全画像で取得 |
| 3 | リザルト画面の判定数（COOL/GREAT/GOOD/BAD）の取得 | 可否を明らかにする |
| 4 | プレイ画面 1枚あたりの処理時間 | ≤ 200ms |
| 5 | リザルト画面 1枚あたりの処理時間 | ≤ 200ms |

## 3. 実施内容

### 3.1. ROI 定義（1366×768 基準）

プレイ画面:
- `song_name`: x=(350, 980), y=(0, 65) — 上部黒帯白文字
- `score`: x=(110, 410), y=(590, 665) — 左下スコア

リザルト画面:
- `score`: x=(680, 880), y=(415, 470)
- `cool`: x=(740, 880), y=(472, 510)
- `great`: x=(740, 880), y=(500, 538)
- `good`: x=(740, 880), y=(528, 565)
- `bad`: x=(740, 880), y=(558, 595)
- `combo`: x=(740, 880), y=(600, 640)
- `best_score`: x=(740, 880), y=(640, 680)
- `best_diff`: x=(740, 880), y=(670, 705)

### 3.2. 楽曲特定アルゴリズム

OCR 結果 → rapidfuzz.process.extractOne（scorer=WRatio）でマスタ
`['ぽぽぽかレトロード', '漆黒のスペシャルプリンセスサンデー']` と照合。
WRatio ≥ **65** で特定確定（初版は70だったが調整）。

### 3.3. 数字補正

OCR 結果に含まれる全角数字・「Ｓ→5」「Ｂ→8」「Ｏ→0」等の装飾誤読を半角数字に正規化してから `isdigit()` で抽出。

|------|---------------|-----------|--------|------|

| 指標 | 結果 | 合格基準 | 判定 |

| COMBO 取得 | 3/6 | — | 一部 |
| COOL/GREAT/GOOD/BAD 取得 | **ほぼ全滅（PaddleOCR で検出できず）** | — | ❌ |

#### 判定数の取得失敗の原因分析

クロップ画像（`poc/results/roi_crops/` 配下）を目視確認した結果:

- 数字自体は **正しく切り出されている**（COOL=384、GREAT=218、GOOD=65、BAD=100 など）
- しかし PaddleOCR の text_detection モデルが **色付き太字装飾フォントの単独数字を「テキスト」として検出してくれない**
- v1（ROI 狭い）の段階では一部「3」「218」など部分的に取れていたが、ROI を広げた v2 では全滅
- ROI を狭めすぎても広げすぎても認識できない不安定さがある

**結論**: 判定数（色付き太字数字）は PaddleOCR では信頼性ある取得が困難。**前処理（拡大・グレースケール化・コントラスト強調）または別方式が必要**。

## 5. 採用判定

### 5.1. 確定事項

1. **領域限定OCRは合格基準を満たす**（プレイ画面 95ms、リザルト画面 115ms）。OCRエンジンとして PaddleOCR 2.7.3 を本採用する。
2. **プレイ画面の楽曲名識別は実効100%**（マスタファジーマッチ閾値65）。
3. **プレイ画面のスコアは PaddleOCR で十分**（信頼度 0.96–0.998）。
4. **リザルト画面の SCORE・COMBO・BEST SCORE は PaddleOCR で取得可能**。

### 5.2. 詳細設計で対応すべき事項

| ID | 内容 | 対応方針 |
|----|------|----------|
| F-1 | リザルト画面の判定数（COOL/GREAT/GOOD/BAD）認識 | **方式 A**: 数字テンプレートマッチング（pop'n 固定フォント数字テンプレ画像を作成し、画素単位で最大一致を取る）。**方式 B**: 前処理（色マスク→グレースケール→3倍拡大→コントラスト強調）後に PaddleOCR 再実行。詳細設計工程で両方式を比較 PoC #03 で実施 |
| F-2 | マスタファジーマッチング閾値 | 実マスタ（数百〜千件規模）で再評価。65 は本 PoC 仮値。誤マッチ抑止のためジャンル名・難易度ラベルとの一致でスコアブーストを行う設計（基本設計 §4.5）と組み合わせて閾値を決定 |
| F-3 | 楽曲名OCR の「ぽ」→「ぼ」「ほ」誤読対策 | rapidfuzz の文字置換テーブルに追加する／PaddleOCR の rec モデルファインチューニング（コスト高、見送り想定）／マスタ側の正規化辞書で対応 |
| F-4 | 初回フレームの処理時間（300ms超） | 起動時にウォームアップ推論を1回実行してキャッシュを温める設計を入れる |
| F-5 | リザルト画面の BEST SCORE 行は2行に分かれる | `best_score` と `best_diff` を別ROIにする ROI 定義を採用（v2で対応済み） |

### 5.3. PoC #03 の必要性

判定数（F-1）の方式比較を **PoC #03（数字認識方式選定）** として詳細設計フェーズ冒頭に追加実施する。スコープ:
- 入力: リザルト画面サンプル 6枚 × 4判定 = 24枚の数字画像
- 比較: テンプレートマッチング vs 前処理+PaddleOCR
- 評価: 認識正答率、処理時間
- 期限: 詳細設計の DB/UI 設計開始前まで
