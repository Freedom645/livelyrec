## 1. データベース基本方針

- DBMS: **SQLite 3.40+**（Python 3.11 標準同梱）
- ファイル配置: `<配布フォルダ>/livelyrec_data/db/livelyrec.sqlite3`（ポータブル構成、`05_基本設計書.md` §9.1 参照）
- 文字コード: UTF-8（SQLite 既定）
- 外部キー: `PRAGMA foreign_keys = ON;` を接続ごとに必ず ON
- ジャーナルモード: `PRAGMA journal_mode = WAL;`（配信中の同時読書きを安定化）
- 同期モード: `PRAGMA synchronous = NORMAL;`（性能と耐障害性のバランス）
- 単一接続前提（複数スレッドからの同時書込みは行わない設計）

## 2. DDL 全体

```sql
-- 楽曲マスタ
CREATE TABLE song (
    song_id        TEXT PRIMARY KEY,            -- 安定 ID（例: "popn-12345"）
    title          TEXT NOT NULL,               -- 表示用タイトル
    title_norm     TEXT NOT NULL,               -- 検索・マッチング用正規化済み
    genre          TEXT,
    has_upper      INTEGER NOT NULL DEFAULT 0,  -- 0/1
    fetched_at     TEXT NOT NULL                -- ISO 8601
);
CREATE INDEX idx_song_title_norm ON song(title_norm);

-- 譜面（楽曲×難易度）
CREATE TABLE chart (
    chart_id    TEXT PRIMARY KEY,               -- "{song_id}:{difficulty}:{upper}"
    song_id     TEXT NOT NULL REFERENCES song(song_id) ON DELETE CASCADE,
    difficulty  TEXT NOT NULL CHECK (difficulty IN ('EASY','NORMAL','HYPER','EX','UPPER')),
    is_upper    INTEGER NOT NULL DEFAULT 0,
    level       INTEGER,                        -- 難易度値（不明時は NULL）
    UNIQUE (song_id, difficulty, is_upper)
);
CREATE INDEX idx_chart_song ON chart(song_id);

-- プレイセッション
CREATE TABLE play_session (
    session_id     TEXT PRIMARY KEY,            -- UUID4 hex
    chart_id       TEXT NOT NULL REFERENCES chart(chart_id) ON DELETE RESTRICT,
    started_at     TEXT NOT NULL,               -- ISO 8601
    ended_at       TEXT,                        -- ISO 8601, NULL なら未終了
    business_date  TEXT NOT NULL,               -- "YYYY-MM-DD"（業務日）
    attempt_count  INTEGER NOT NULL DEFAULT 1,
    final_status   TEXT NOT NULL CHECK (final_status IN
                       ('IN_PROGRESS','COMPLETED','SKIPPED','RETRIED_OUT','ABANDONED')),
    obs_scene      TEXT,                        -- 取得時の OBS シーン名
    obs_source     TEXT,                        -- 取得時のソース名
    resolution     TEXT                          -- "WxH"
);
CREATE INDEX idx_session_business_date ON play_session(business_date);
CREATE INDEX idx_session_chart ON play_session(chart_id);

-- リザルト（プレイ1回ごとに最大1件）
CREATE TABLE result (
    session_id        TEXT PRIMARY KEY REFERENCES play_session(session_id) ON DELETE CASCADE,
    score             INTEGER NOT NULL CHECK (score BETWEEN 0 AND 100000),
    cool              INTEGER NOT NULL CHECK (cool >= 0),
    great             INTEGER NOT NULL CHECK (great >= 0),
    good              INTEGER NOT NULL CHECK (good >= 0),
    bad               INTEGER NOT NULL CHECK (bad >= 0),
    combo             INTEGER NOT NULL CHECK (combo >= 0),
    clear_type        TEXT NOT NULL CHECK (clear_type IN
                          ('PERFECT','FULL_COMBO','CLEAR','FAILED')),
    medal             TEXT NOT NULL CHECK (medal IN
                          ('STAR_GOLD','STAR_SILVER','STAR_BRONZE',
                           'DIAMOND_GOLD','DIAMOND_SILVER','DIAMOND_BRONZE',
                           'CIRCLE','NONE')),
    rank              TEXT NOT NULL CHECK (rank IN
                          ('S+','S','AAA','AA+','AA','A+','A','B+','B','C+','C','D','E')),
    best_score_diff   INTEGER,                  -- BEST SCORE 差分（NULL 可）
    recorded_at       TEXT NOT NULL
);
CREATE INDEX idx_result_score ON result(score);

-- リトライ
CREATE TABLE retry (
    retry_id     TEXT PRIMARY KEY,              -- UUID4
    session_id   TEXT NOT NULL REFERENCES play_session(session_id) ON DELETE CASCADE,
    occurred_at  TEXT NOT NULL
);
CREATE INDEX idx_retry_session ON retry(session_id);

-- 業務日メタ
CREATE TABLE business_day (
    business_date  TEXT PRIMARY KEY,            -- "YYYY-MM-DD"
    rolled_at      TEXT NOT NULL                -- 切替が実際に走った時刻
);

-- 業務日打鍵集計
CREATE TABLE daily_keycount (
    business_date  TEXT NOT NULL REFERENCES business_day(business_date) ON DELETE CASCADE,
    judgement      TEXT NOT NULL CHECK (judgement IN ('COOL','GREAT','GOOD','BAD','TOTAL')),
    count          INTEGER NOT NULL DEFAULT 0,
    updated_at     TEXT NOT NULL,
    PRIMARY KEY (business_date, judgement)
);

-- アプリ設定（KV）
-- センシティブな値は infrastructure/config_store.py で別途 DPAPI 管理。
-- 本テーブルは小規模な内部状態を保持する。
CREATE TABLE app_kv (
    key    TEXT PRIMARY KEY,
    value  TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

-- スキーマバージョン
CREATE TABLE schema_version (
    version    INTEGER PRIMARY KEY,
    applied_at TEXT NOT NULL,
    note       TEXT
);
INSERT INTO schema_version (version, applied_at, note) VALUES (1, datetime('now'), 'initial');
```

## 3. テーブル仕様の補足

### 3.1. `song` / `chart`

- `song_id` は **マスタJSONで採番された安定ID**を採用。LivelyRec 側で自動採番しない。
- `title_norm` は `domain/master.py` の正規化関数で生成（ひらがな化・全半角統一・記号除去）。マッチング時のキー。
- UPPER譜面は同一 song の chart 行として持つ（`is_upper=1, difficulty='UPPER'`）。

### 3.2. `play_session`

- セッションは「プレイ画面突入」で生成、「リザルト遷移 or 状態リセット」で確定。
- `attempt_count` はリトライ検知時にインクリメント。
- `final_status`:
  - `IN_PROGRESS`: プレイ中
  - `COMPLETED`: リザルト画面まで到達し記録完了
  - `SKIPPED`: リザルトをスキップされ詳細記録なし
  - `RETRIED_OUT`: プレイ画面で別譜面に切り替わった
  - `ABANDONED`: アプリ停止等で未確定のまま終了

### 3.3. `result`

- リザルト画面でのみ INSERT。スキップ時は本テーブルに行が無い。
- `medal` / `rank` は **記録時に算出して保存**（後方互換のため）。算出ロジックは `10_詳細設計_画像認識.md` §6 参照。
- `best_score_diff` は OCR で取れた場合のみ。取れない場合 NULL。

### 3.4. `retry`

- プレイ画面 → プレイ画面（リトライ検出）時に INSERT。
- セッション削除でカスケード削除。

### 3.5. `business_day` / `daily_keycount`

- 業務日は **AM 6:00 切替**（設定可、既定）。
- ロール時刻 (`rolled_at`) を保存しておくことで後の検証が容易。
- 4種類の判定 + TOTAL を行として持つ（クエリ簡略化のため冗長保持）。

### 3.6. `app_kv`

- 例: 最後のクラッシュ時刻、最終マスタ取得時刻、UI ペインのサイズなど。

## 4. リポジトリインターフェイス

`infrastructure/repository/*.py`。サンプルとして `result_repo.py`:

```python
class ResultRepository:
    def __init__(self, conn: sqlite3.Connection) -> None: ...

    def upsert(self, session_id: str, result: Result, recorded_at: datetime) -> None: ...

    def get_by_session(self, session_id: str) -> Result | None: ...

    def list_by_chart(self, chart_id: str, limit: int = 20) -> list[tuple[PlaySession, Result]]: ...

    def best_score(self, chart_id: str) -> int | None: ...
```

各リポジトリは **単一接続を共有** し、内部でトランザクションを張る。

## 5. マイグレーション戦略

- 起動時に `schema_version` を読み、`migrate.py` の昇順マイグレーションを順次適用。
- 初版（v1）以降の変更は `migrations/0002_*.sql` のように番号付き SQL ファイルで管理。
- ダウングレード（rollback）は提供しない。

```python
def migrate(conn: sqlite3.Connection, target: int = LATEST) -> None:
    current = _read_current_version(conn)
    for v in range(current + 1, target + 1):
        sql = _load_migration(v)
        conn.executescript(sql)
        conn.execute("INSERT INTO schema_version VALUES (?, datetime('now'), ?)", (v, ...))
        conn.commit()
```

## 6. クエリパターン（性能）

### 6.1. 主要クエリ

| 用途 | クエリ | 利用インデックス |
|------|--------|------------------|
| 過去リザルト N 件取得（譜面指定） | `SELECT * FROM result JOIN play_session USING(session_id) WHERE play_session.chart_id = ? ORDER BY play_session.started_at DESC LIMIT N` | `idx_session_chart` |
| 当日業務日の累計打鍵 | `SELECT judgement, count FROM daily_keycount WHERE business_date = ?` | PK |
| 譜面別ベストスコア | `SELECT MAX(score) FROM result JOIN play_session USING(session_id) WHERE play_session.chart_id = ?` | `idx_session_chart` |
| 楽曲名サジェスト | `SELECT * FROM song WHERE title_norm LIKE ? LIMIT N` | `idx_song_title_norm` |

### 6.2. データ量見積もり

- 楽曲数: 〜2000
- 譜面数: 〜10000
- プレイセッション: 1日100セッション × 1年 = 36500行 → 5年で 18万行
- リザルト: ほぼセッション数と同じ
- daily_keycount: 1日5行 × 5年 = 9125行

→ 全部合わせても数十MB。SQLite で十分。

## 7. バックアップ・復元

- アプリ自動バックアップ機能は v1.0 で提供しない（NFR-OPS-003 範囲外）。
- ユーザは `livelyrec.sqlite3` ファイルをコピーして任意の場所に保管可能。
- 復元はファイルを置き換えるだけ。
