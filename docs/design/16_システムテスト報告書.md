## 3. 区分A: 録画リプレイテスト結果

### 3.1. 画面判別

| 対象 | 結果 |
|------|------|
| `tests/fixtures/sample/` 代表フレーム | 41/41 = **100%** |
| 実 footage 474 フレーム（ハイブリッド改修後） | 464/474 = **97.9%** |

右下 HSV シグネチャに、画面全体サムネイル照合によるタイトル/クエスト判定を前段追加。

### 3.2. リザルト認識精度

正解ラベル付きリザルト **12 画面**（`_ground_truth.csv`）で測定:

| 項目 | 結果 |
|------|------|
| スコア | **100%** |
| コンボ | 92% |
| 判定数 COOL/GREAT/GOOD/BAD | ほぼ 100%（cool のみ 92%） |

- digit テンプレートを 2 系統化（スコア用＝橙特化二値化／combo・判定数用）。
- スコアの色マスクを橙特化（H 10–30・高彩度・高明度でグレア除外）し、スコア 100% を達成。

## 4. 認識ロジックの改善

| 改善 | 内容 |
|------|------|
| digit テンプレート 2 系統化 | スコア用（橙特化二値化）と combo/判定数用を分離・再生成 |
| 画面判別ハイブリッド化 | 右下シグネチャ＋タイトル/クエスト参照照合 |
| クリアランク | 仕様確定（8 種・クリア成否依存）で `clear_rank` を再実装 |
| 難易度判定 | プレイ画面の難易度バッジのテーマ色（EASY/NORMAL/HYPER/EX）を最近傍照合し譜面を特定 |
| 楽曲タイミング修正 | 非プレイ画面で楽曲特定状態をリセットし、各楽曲をプレイ中に新規特定。前楽曲の持ち越しを解消 |

## 5. 区分B: 実環境システムテスト

連続稼働テストを実施し、**クラッシュ・例外なし**。観測した 9 点の対応状況:

| # | 指摘 | 対応 |
|---|------|------|
| 1 | リザルトスコア認識精度が低い | ✅ スコア専用テンプレート＋橙二値化 → スコア 100% |
| 2 | 直近リザルトに楽曲名が出ない | ✅ `result.recorded` に title 追加・表示 |
| 3 | クリアランクが誤り | ✅ 仕様確定（8 種・クリア成否依存）で再実装 |
| 4 | 現在状態の楽曲/難易度が反映されない | ✅ 難易度バッジ判定を追加。`play.started` に楽曲・難易度を反映 |
| 5 | 打鍵数カウンタが動かない | ✅ `judgements.tick` 発火を追加しリザルト確定で正確に更新。※プレイ中のライブ追従は下記のとおり v1 スコープ外 |
| 6 | 「業務日」の用語 | ✅ 「プレイ日」へ変更（UI・要件文書） |
| 7 | スコアが1つ前の楽曲に紐づいて記録される | ✅ 楽曲特定状態のリセット修正で解消 |
| 8 | 配信支援HTMLオーバーレイが現在状態に同期しない | ✅ 真因は LAN公開・トークン認証時にブラウザがヘッダ認証できず接続拒否→再接続ループ。URLクエリ `?token=` 認証を追加。あわせて WS サーバが最新イベントを保持し接続直後に再送 |
| 9 | 起動直後、最初のリザルトまで打鍵数が表示されない | ✅ 起動時に日次カウンタの初期スナップショットを通知 |

### 打鍵数ライブ追従（#5）を v1 スコープ外とした理由

プレイ画面下部の判定数表示は、(1) 数字どうしの接触、(2) 数字と同色の背景バー、(3) 暗色の先頭ゼロ、(4) リザルトと異なるフォント、(5) 1の位に重なる丸角枠 ── の 5 障害が重複し、安定した per-frame 抽出が困難。日次カウンタはリザルト確定値（認識精度高）で**曲ごとに正確に更新**されるため、ライブ追従は将来の改善課題とする。
