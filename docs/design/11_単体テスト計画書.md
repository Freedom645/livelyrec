## 2. テスト対象範囲

### 2.1. 対象（In-Scope）

純粋ロジック（外部プロセス・実機ハードウェア・学習済みモデルに依存しない部分）を単体テストの対象とする。

- domain 層：状態機械・スコア計算・業務日・ランク/メダル・マスタ照合の全ロジック
- application 層：マスタ照合サービス、アップデート判定、画面分析、CSV出力、記録要否判定
- infrastructure/repository 層：SQLite リポジトリの CRUD・マイグレーション
- infrastructure/recognizer 層：フレーム正規化・ROI 抽出・画面判別・リトライ検出・パイプライン分岐
- infrastructure/ocr 層のうち **数字テンプレートマッチング**（`digit_template.py`）：OpenCV のみで完結し合成画像で検証可能
- shared 層：定数・例外・業務日計算・パス解決・ロギング構成

### 2.2. 対象外（Out-of-Scope / 後工程へ委譲）

下記は実機・実通信・学習済みモデルに依存し、単体テストでの完全検証が不適当なため、**スモーク確認（構築・呼び出しが例外なく通ること）のみ**を本工程で行い、本格的な検証は結合テスト・システムテストに委譲する。

| モジュール | 委譲理由 | 委譲先 |
|------------|----------|--------|
| `infrastructure/obs_client.py` | OBS Studio 実プロセス・WebSocket 実通信が必要 | 結合テスト |
| `infrastructure/ws_server.py` | 実 WebSocket 接続・ブラウザソース連携が必要 | 結合テスト |
| `infrastructure/ocr/paddle.py` | PaddleOCR 学習済みモデルの実ロードが必要 | 結合テスト |
| `application/recording_service.py` のスレッド駆動メインループ | OBS取得→分析→記録の連続稼働検証が本質 | 結合テスト |
| `ui/` 配下の全ウィジェット・`main_window.py` | 実描画・ユーザ操作が本質。本工程は pytest-qt による構築スモークのみ | システムテスト |
| `app.py` の起動シーケンス | QApplication 実起動が必要。本工程は DI 組立関数のスモークのみ | システムテスト |

> `recording_service.py` は、記録要否判定・セッション確定などの**純ロジック関数は本工程の対象**とし、それらを呼び出すスレッドループ本体のみを委譲する（§3 の個別目標を参照）。

## 4. テスト方針・観点

### 4.1. 基本方針

- テストフレームワークは `pytest`。GUI スモークは `pytest-qt`。カバレッジ計測は `pytest-cov`。
- 1テスト1観点を原則とし、テスト名で検証内容が判別できる命名にする。
- 外部依存（OBS・HTTP・WebSocket・PaddleOCR モデル）は `monkeypatch` / フェイク実装 / `tmp_path` で代替する。
- 画像処理テストは KONAMI 公式アセットを使わず、**numpy で合成した HSV/BGR 画像**を用いる（NFR-LEGAL-001 遵守）。サンプル画像が必要な回帰テストは `tests/fixtures/sample/` の既存収集分のみ使用。
- SQLite は `tmp_path` 上の一時 DB を用い、テスト間の独立性を保つ。

### 4.2. テスト観点

| 観点 | 内容 |
|------|------|
| 正常系 | 設計どおりの入力で期待出力が得られる |
| 境界値 | 業務日切替時刻、カバレッジ閾値、コンボ0、判定数0、空文字列など |
| 異常系 | 不正入力・欠損データ・例外発生時に定義どおりの例外/フォールバックになる |
| 状態遷移 | StateMachine の不正遷移3連続観測許容、リトライ検出の不可逆遷移 |
| 回帰 | 画面判別・ROI 抽出は既存サンプル画像で過去結果を固定 |

### 4.3. 重点検証項目（必須機能＝スコア記録に直結）

- StateMachine：画面遷移の妥当性検証と不正遷移デバウンス
- リトライ検出：「コンボ0かつ全判定0への不可逆遷移」（FR-REC-037）
- 画面判別：右下シグネチャ方式での7画面分類、OPTION/RESULT のテンキー点灯分離
- マスタ照合：ファジーマッチング閾値、SongStabilizer 多数決
- ランク/メダル算出：境界スコアでの区分
- リポジトリ：プレイセッション・リザルトの登録/取得の整合

## 7. 合否判定基準・完了条件

工程6は以下を**すべて**満たした時点で完了とし、`12_単体テスト報告書.md` に結果を記録、ユーザ（PO）レビュー承認をもって工程7へ移行する。

1. §3.1 のレイヤ別カバレッジ目標を**全レイヤで達成**（`recording_service.py` は §3 の個別ルールで評価）。
2. 致命欠陥（Critical）が **0 件**（§8 参照）。
3. 全自動テストがパスする（`pytest` 実行で失敗 0）。
4. `ruff check` がテストコードを含めてパスする。
5. 「対象外」モジュールのスモークテストがすべて通過する。

## 8. 致命欠陥の定義と扱い

| 等級 | 定義 | 工程完了への影響 |
|------|------|------------------|
| 致命（Critical） | スコア記録の中核機能の誤動作、データ欠損・破損、無条件クラッシュ | **0 件必須**。残存時は工程完了不可、実装へ差戻し |
| 重大（Major） | 主要機能の一部不具合、限定条件下の誤動作 | 報告書に記載。原則修正、PO 判断で次工程持越し可 |
| 軽微（Minor） | 表示崩れ・軽微なメッセージ不備など影響限定 | 報告書に記載。リスク・課題管理表へ登録のうえ持越し可 |

発見した欠陥は `04_リスク・課題管理表.md` の課題一覧（`I-xxx`）に登録して追跡する。

## 10. リスク・前提

- **R-013 / R-022**：サンプル画像が少量で、画像認識モジュールの実機網羅は限定的。本工程は合成画像中心で論理分岐を網羅し、実画像での精度検証はシステムテスト（実プレイ録画）で行う。
- **R-022**：数字テンプレート「7」「9」未生成。`digit_template.py` のテストは合成テンプレで論理を検証し、実テンプレ依存部分はシステムテストに委譲する。
- 「対象外」モジュール（OBS/WS/PaddleOCR/UI）の本格検証は後工程に依存するため、本工程の完了は実機検証の完了を意味しない。
