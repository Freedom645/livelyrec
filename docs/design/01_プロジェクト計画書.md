## 1. プロジェクトの目的

pop'n music lively（KONAMI Amusement のコナステ配信タイトル）のプレイ画面を OBS Studio の WebSocket 経由でキャプチャ取得し、画像認識によりスコアおよびプレイ統計を自動記録するスタンドアロン Windows アプリ「**LivelyRec**」を開発する。あわせて OBS ブラウザソースから利用可能な配信支援表示（打鍵数グラフ・過去リザルト表示）を提供する。

### 1.1. 解決したい課題

- 手動でのスコア記録は煩雑で、配信中の集中を阻害する。
- 公式サービスは譜面別の詳細リザルト履歴を長期に保管しない／参照が不便。
- 配信演出（打鍵カウンタ・過去スコア表示）を作るには個別実装が必要で、配信者ごとに労力が分散している。

### 1.2. 達成目標（KGI/KPI 相当）

| 観点 | 目標 |
|------|------|
| 機能達成 | 必須機能（スコア記録）の全要件が受入基準を満たす |
| 精度 | プレイ画面・リザルト画面でのスコア/判定数/コンボ数の認識正答率 ≥ 99%（標準解像度のキャプチャ・標準テーマ条件下） |
| 安定性 | 1配信（連続8時間）でクラッシュ・データ欠損なしを目標 |
| 配信支援 | OBS ブラウザソース表示の更新遅延 ≤ 500ms（プレイ画面の判定数更新からブラウザ反映まで） |

## 2. スコープ

### 2.1. 対象スコープ（In-Scope）

- pop'n music lively のゲーム画面の取得と画面種別判定（選曲／準備／オプション／プレイ／リザルト）
- プレイ画面での打鍵数カウント、リザルト画面でのスコア・判定数・コンボ・クリア種類記録
- マスタデータ（楽曲名・難易度）との照合による楽曲特定
- クリアメダル・クリアランクのアプリ内算出
- OBS WebSocket 経由のキャプチャ取得
- OBS ブラウザソース向けの配信支援表示（HTML/JS）
- 自動アップデート（GitHub Releases 連携）
- CSV エクスポート
- 外部ツール連携（WebSocket API 公開・データファイル書き出し）

### 2.2. 対象外（Out-of-Scope）

- pop'n music lively 以外のシリーズ作品（ピーストリエ／UniLab／解禁版／旧筐体筐体向けアーケード版等）対応
- Mac / Linux サポート
- スマートフォン／タブレット端末からの利用
- 他プレイヤーとのオンライン共有・SNS自動投稿
- ゲームソフト本体やKONAMI公式APIへの直接アクセス
- 不正な手段（メモリ読み取り等）でのデータ取得

### 2.3. 前提条件

- 利用 PC は Windows 10 64bit 以降。GPU 要件は OBS Studio の動作要件に準ずる。
- ユーザは pop'n music lively を正規プレイ可能な環境（コナステ）を保有している。
- OBS Studio 28 以降（WebSocket v5 同梱版）が利用可能。
- ネットワーク接続が常時利用可能（マスタデータ取得・アップデートチェック）。

### 2.4. 制約事項

- KONAMI の商標・著作物の権利を侵害しない範囲で開発する（公式が配布する画像アセット等をリポジトリに同梱しない）。
- 規約上問題のある自動化（自動プレイ、データ改竄、サーバへの不正アクセス）は実装しない。

## 7. 技術スタック

要件定義工程末で **Python + PySide6 + OpenCV + PaddleOCR** に確定（2026-05-18 承認）。本節では確定内容と、検討時に比較した代替案を記録として残す。

### 7.1. 採用構成（確定）

| 区分 | 採用ライブラリ／フレームワーク | 用途 |
|------|--------------------------------|------|
| 言語 | Python 3.11 以上 | アプリ本体 |
| GUI | PySide6（Qt for Python） | デスクトップUI |
| 画像処理 | OpenCV（cv2） | 画面判別・前処理・テンプレートマッチング |
| OCR | PaddleOCR（第一候補） / Tesseract（代替候補） | 楽曲名・スコア等の文字認識。基本設計PoCで最終確定 |
| OBS連携 | `obs-websocket-py`（OBS WebSocket v5 クライアント） | ゲーム画面キャプチャ取得 |
| 外部WebSocket | `websockets` ライブラリ | 外部ツール／配信支援ブラウザソース連携 |
| 永続化 | SQLite（標準ライブラリ `sqlite3`） | スコア・累計データの永続化 |
| パッケージング | PyInstaller | 単一EXE / portable zip 配布 |
| 自動更新 | 自前実装（GitHub Releases API） | バージョンチェック・適用 |
| 設定保護 | Windows DPAPI（`pywin32` 経由） | OBSパスワード等の保護 |
| 静的解析 | ruff、mypy | コーディング規約・型チェック |
| テスト | pytest、pytest-qt | ユニット・GUIテスト |

### 7.2. 候補比較（採用判断の根拠として記録）

| 観点 | A. Python + PySide6 | B. C# / .NET 8 + WPF | C. Electron + TypeScript |
|------|---------------------|-----------------------|--------------------------|
| 画像認識ライブラリ | OpenCV(cv2)、Pillow、Tesseract(pytesseract)、ONNX Runtime ── **業界標準・サンプル豊富** | OpenCvSharp、Emgu CV、Tesseract.NET、Windows.Media.Ocr ── **十分だが日本語OCR精度は要評価** | opencv.js、Tesseract.js ── **JS化されており性能は劣る** |
| OBS WebSocket | `obs-websocket-py`（公式系）、`simpleobsws` ── 安定 | `OBSWebsocketDotNet` ── 安定 | `obs-websocket-js`（公式） ── 安定 |
| GUI 構築コスト | 中（PySide6 で十分作れる） | 中〜低（XAML資産・デザイナ） | 低（HTML/CSS そのまま） |
| 配信支援ブラウザソース | HTML/JS を別途用意 | 同左 | **同一スタックで実装可能（強み）** |
| Windows 配布 | PyInstaller、Briefcase など。コード署名は別途 | 単一exe発行容易、MSIX対応も実績多 | electron-builder、コード署名対応 |
| 自動アップデート | 自前実装 or `pyupdater`（メンテ低調） | `Squirrel.Windows` / `Velopack` | `electron-updater`（成熟） |
| パッケージサイズ | 中（〜150MB） | **小（〜80MB、AOT可）** | 大（〜200MB+） |
| 起動速度 | 中 | **速い** | 中〜遅 |
| エコシステム成熟度 | 画像認識領域では最強 | Windowsデスクトップでは最強 | Webデスクトップ全般で最強 |
| 学習・着手コスト | 低（プロトタイピング容易） | 中（XAML/MVVM） | 低（Web知識でカバー） |
| 主リスク | exe配布時の起動速度・サイズ、コード署名 | 画像認識の日本語OCRライブラリ選択 | パッケージサイズ、CV処理性能 |

### 7.3. 採用理由

1. **画像認識領域のエコシステム最強**: pop'n の楽曲名・スコア・判定数認識は本アプリの心臓部であり、Python のエコシステム（OpenCV、PaddleOCR、Tesseract、機械学習資産）が最も豊富。PoC〜本実装まで同一言語で完結する。
2. **プロトタイピング速度**: 基本設計フェーズで画像認識方式PoCを実施する想定。Python の REPL/Jupyter での試行錯誤が圧倒的に速い。
3. **GUI / WebSocket / 自動アップデート / CSV のいずれも枯れたOSSで対応可能**。
4. **パッケージサイズ・起動速度**は本アプリのユースケース（配信中常駐）では致命傷ではない。

### 7.4. 不採用案の記録

| 案 | 不採用理由 |
|----|-----------|
| C# / .NET 8 + WPF | Windows ネイティブの品質は高いが、画像認識（特に日本語OCR）でPythonほどの選択肢がなく、PoCコストが上回ると判断 |
| Electron + TypeScript | 配信支援UIとの同一スタック化は魅力だが、CV処理性能とパッケージサイズが本ユースケースに不利 |

## 8. 品質方針

- **コード品質**: 静的解析（Pythonなら ruff / mypy）と CI による Lint/型チェック。
- **テスト方針**:
  - 単体テスト: 画像認識アルゴリズム（画面判別・OCR）にはサンプル画像（`tests/fixtures/sample/` 既存）を用いた回帰テストを必須化。
  - 結合テスト: OBS WebSocket と本アプリの疎通、ブラウザソースとの WebSocket 接続。
  - システムテスト: 実プレイ録画を用いたエンドツーエンド検証。
- **ログ**: ファイルローテーション付きロガー。配信中の障害解析を可能にする粒度。
