## 1. 目的と位置づけ

- 実プレイ環境での**エンドツーエンド検証**を行い、製品が要求品質を満たすことを確認する。
- プロジェクト計画書 §1.2 が定める KGI/KPI の達成を確認する:
  - 認識正答率 **≥ 99%**（プレイ/リザルト画面のスコア・判定数・コンボ、標準解像度・標準テーマ条件下）
  - 1配信相当（**≥ 1 時間**）の連続稼働でクラッシュ・データ欠損なし
  - 配信支援表示の更新遅延 **≤ 500ms**（NFR-PERF-003）
- 工程7「結合テスト」§7 の申し送り（楽曲名 OCR 精度、画面誤判定、clear_type 判定、プレイ combo 認識、7/9 テンプレ補強）を本工程で対応する。
- 本工程は受入前の最終品質ゲートである。

## 2. テスト対象範囲

### 2.1. 対象（In-Scope）

| 区分 | 内容 |
|------|------|
| 認識精度 | プレイ画面（楽曲名・スコア・コンボ）、リザルト画面（スコア・判定数・コンボ・clear_type）、画面判別 |
| 連続稼働 | ≥ 1 時間の連続記録でのクラッシュ・データ欠損・メモリ使用量の検証 |
| 配信支援 | OBS ブラウザソース表示の内容反映と更新遅延 |
| ドメイン挙動 | リトライ検出、業務日切替、CSV エクスポートのエンドツーエンド動作 |

### 2.2. 対象外（Out-of-Scope）

- 単体テスト・結合テストで検証済みの個別モジュール挙動の再検証。
- pop'n music lively 以外のタイトル、Mac/Linux（要件 §2.2 のとおり）。

## 3. テスト区分

実プレイ環境はユーザの実機に依存するため、結合テストと同様に2区分で構成する。

- **区分A 録画リプレイテスト**: 収集した実プレイフレーム（正解ラベル付き）に対し認識パイプラインを通し、認識結果と正解を突合して**認識正答率を測定**する。スクリプト／pytest で自動化し、Claude Code が実施する。
- **区分B 実環境システムテスト**: 実プレイでの連続稼働・配信支援表示・エンドツーエンド動作を、ユーザが実機環境で実施する。手順書化する。

---

## 4. 認識精度の測定（区分A）

### 4.1. 目標（KPI）

| 項目 | 目標正答率 |
|------|-----------|
| 画面判別 | ≥ 99% |
| リザルト スコア／判定数／コンボ | ≥ 99% |
| プレイ画面 スコア | ≥ 99% |
| 楽曲名の楽曲特定（マスタ照合後） | ≥ 99% |

> 楽曲名は OCR 単体ではなくマスタファジーマッチング後の**楽曲特定の正否**で評価する（OCR 誤読はマスタ照合で吸収される設計のため）。

### 4.2. 測定方法

1. 正解ラベル付きの実プレイフレーム集を用意する（§4.3）。
2. 各フレームを `RecognitionPipeline` に通し、認識結果を得る。
3. 認識結果と正解ラベルを項目別に突合し、正答率を算出する。
4. 未達項目は認識ロジック（ROI・色マスク・テンプレート・画面判別しきい値）を改善し、再測定する。

### 4.3. テストデータの収集

- **デバッグ撮影機能**（`livelyrec_data/debug/`、設定「記録」タブで ON）で実プレイ中のフレームを収集する。
- 各画面**30 枚以上**を目標とする（R-013）。数字「7」「9」を含むリザルトを必ず含める（R-022）。
- 収集フレームに正解ラベルを付与する（ファイル名規約、または対応表 CSV）。
- 既存の `tests/fixtures/sample/` 画像も画面判別の正解データとして用いる。

---

## 5. 連続稼働テスト（区分B）

- 実プレイで **≥ 1 時間**（目標は1配信相当）の連続記録を行う。
- 確認項目:
  - クラッシュ・フリーズが発生しないこと。
  - 記録データの欠損・重複がないこと。
  - メモリ使用量が増加し続けないこと（リーク兆候の有無）。
  - 業務日切替（既定 AM6:00）が正しく動作すること（必要に応じ切替時刻を調整して検証）。
  - 配信支援ブラウザソースの表示が記録に追従し、更新遅延が体感 ≤ 1 秒であること。

## 7. 合否判定・完了条件

工程8 は以下を満たした時点で完了とし、`16_システムテスト報告書.md` に結果を記録、PO レビュー承認をもって工程9「受入」へ移行する。

1. 区分A の認識正答率が §4.1 の KPI（≥ 99%）を達成。**または**未達項目が許容範囲であることを PO が承認。
2. 区分B の連続稼働（≥ 1 時間）でクラッシュ・データ欠損が無い。
3. 配信支援表示の更新遅延が ≤ 500ms（体感確認で可）。
4. 致命欠陥（Critical）0 件。
5. 全自動テスト（単体・結合）がパスし、`ruff` がクリーン。

> 認識精度は実環境条件（解像度・テーマ・OBS レイアウト）に依存するため、KPI 未達時は条件を明示のうえ PO 判断で受容可能とする（計画書 §4.2 の反復・受容方針）。

## 8. 作業計画

1. ユーザがデバッグ撮影機能で実プレイフレームを収集し、正解ラベルを付与する。
2. 区分A 録画リプレイテストを実装・実施し、認識正答率を測定する。
3. 未達項目の認識ロジックを改善する（楽曲名 OCR／画面判別／clear_type／プレイ combo／7・9 テンプレ補強）。改善ごとに再測定する。
4. 区分B 実環境システムテストの手順書を整備し、ユーザが連続稼働・配信支援を検証する。
5. 結果を `16_システムテスト報告書.md` に記録し、PO レビューへ。

---

## 9. リスク・前提

- **R-013**: サンプル不足。区分A の前提としてフレーム収集が必須。各画面 30 枚以上を確保する。
- **R-022**: 7/9 数字テンプレートが単一サンプル由来。複数サンプルから再生成して補強する。
- **R-004**: 解像度・OBS レイアウトのバリエーション。標準条件での KPI 達成を主目標とし、非標準条件は制約として明示する。
- **R-007**: KONAMI 側 UI 変更。本工程は現行 UI 前提で検証する。
- **R-016/R-017**: 配信品質（処理負荷・表示遅延）。連続稼働テストで併せて観察する。
- 区分B はユーザの実機環境・実プレイに依存するため、本工程の完了はユーザ実施の完了を要する。
