> 本書は工程7「結合テスト」の計画を定義する。モジュール間および外部システム（OBS Studio・ブラウザソース・PaddleOCR）との結合を検証する。
> 工程6「単体テスト」で後工程へ委譲したモジュール（`obs_client` / `ws_server` / `ocr/paddle` / `recording_service` のメインループ）の検証を本工程で引き取る。

---

## 1. 目的と位置づけ

- 単体で検証済みのモジュールを結合し、**境界をまたぐ連携が設計どおり動作する**ことを確認する。
- 外部システム（OBS Studio・OBS ブラウザソース・PaddleOCR モデル）との疎通を検証する。
- プロジェクト計画書 §4.1 が定める工程7完了条件「**主要シナリオ全件パス**」を満たすことを本工程のゴールとする。

---

## 2. テスト対象範囲

### 2.1. 結合の境界

| 境界 | 内容 |
|------|------|
| OBS ↔ アプリ | OBS WebSocket v5 経由のキャプチャ取得（`obs_client`） |
| 認識パイプライン連結 | `pipeline` → `analysis_service` → `recording_service` → `repository` |
| アプリ ↔ 外部クライアント | `ws_server` ↔ WebSocket クライアント／ブラウザソース |
| マスタ取得連結 | `MasterFetcher` → `MasterService.refresh` → `repository` → `identify` |
| OCR 結合 | `pipeline` ↔ `PaddleOcrEngine`（実モデル） |

### 2.2. テスト区分

本工程の実行環境（CI／開発機）では実 OBS Studio を起動できないため、シナリオを2区分に分ける。

- **自動結合テスト（区分A）**: 外部実機を必要とせず `pytest` で自動実行できるもの。外部システムは境界でフェイク／ローカル実体に差し替える。本工程内で実施する。
- **実環境結合テスト（区分B）**: 実 OBS Studio・実ブラウザ・PaddleOCR 実モデルを要するもの。**手順書として文書化**し、ユーザが実機環境で実施・記録する。結果は `14_結合テスト報告書.md` に記録する。

### 2.3. 対象外

- 実プレイ動画を用いたエンドツーエンドの認識精度検証 → 工程8「システムテスト」
- 長時間連続稼働（1配信相当）の安定性 → 工程8「システムテスト」

## 3. テストシナリオ

### 3.1. 区分A: 自動結合テスト

| ID | シナリオ | 結合範囲 | 合格条件 |
|----|----------|----------|----------|
| IT-WS-01 | WebSocket Server を実バインドし、クライアントが接続して `broadcast` を受信 | `ws_server` ↔ ws クライアント | 送信メッセージがエンベロープ形式で受信される |
| IT-WS-02 | リクエストハンドラ登録 → クライアント要求 → 応答（`*.request`→`*.response`） | `ws_server` ↔ ws クライアント | 登録ハンドラが呼ばれ応答 type が返る |
| IT-WS-03 | 不正 JSON／未知 type の要求に `error` エンベロープを返す | `ws_server` | `error` 応答が返り接続は維持 |
| IT-WS-04 | LAN 公開時のトークン認証：トークン不一致で接続拒否 | `ws_server` | 認証なし接続が code 4401 で切断される |
| IT-REC-01 | フェイク OBS が選曲→プレイ→リザルトの連続フレームを返し、記録ループが回る | `recording_service`↔`analysis`↔`pipeline`↔`repository` | セッション作成・リザルト upsert・日次加算が発生 |
| IT-REC-02 | プレイ中のリトライ検出でリトライが記録される | 同上 | `attempt_count` 増加・`play.retry` 発火 |
| IT-REC-03 | OBS 切断例外 → 再接続シーケンス | `recording_service`↔フェイク OBS | 切断後 `CONNECTING` に遷移し再接続を試行 |
| IT-REC-04 | リザルトのみ観測（プレイ画面見逃し）→ スキップ扱い | `recording_service` | `result.skipped` が発火し記録されない |
| IT-PIPE-01 | 合成フレーム列をパイプラインから分析まで通す | `pipeline`↔`analysis_service` | 画面種別に応じたメトリクスが伝播 |
| IT-MASTER-01 | ローカル JSON を `MasterFetcher` 経由で取得→DB 反映→`identify` | `MasterFetcher`↔`MasterService`↔`repository` | refresh 件数が一致し、identify が譜面を返す |
| IT-MASTER-02 | マスタ取得失敗時にキャッシュへフォールバック | `MasterFetcher` | キャッシュが返るか `MasterFetchError` |
| IT-EXPORT-01 | DB へ記録 → `ExportService` で CSV 出力 → 内容照合 | `repository`↔`ExportService` | 記録件数と CSV 行数が一致 |

> フェイク OBS は `OBSClient` と同じ非同期インターフェイスを実装し、事前に用意した PNG フレーム列（numpy 合成画像を PNG エンコードしたもの）を返す。KONAMI 公式アセットは使用しない。

### 3.2. 区分B: 実環境結合テスト（手順書化）

| ID | シナリオ | 必要環境 | 合格条件 |
|----|----------|----------|----------|
| IT-OBS-01 | 実 OBS Studio へ接続し `GetVersion`・ソース一覧取得 | OBS Studio 28+（WebSocket v5 有効） | 接続成功しバージョン情報が取得できる |
| IT-OBS-02 | 指定ソースのスクリーンショット PNG を取得 | 同上＋キャプチャソース | 1366x768 の PNG が取得できる |
| IT-OBS-03 | OBS 未起動／誤パスワード時のエラー表示 | OBS Studio | 接続パネルにエラーが表示され再試行される |
| IT-BROWSER-01 | ブラウザソース `index.html` を開き WebSocket で受信・表示更新 | ブラウザ（または OBS 内ブラウザ） | 判定数・リザルトが画面に反映される |
| IT-OCR-01 | PaddleOCR 実モデルのロードと楽曲名・数値の OCR | PaddleOCR モデル（初回 DL） | モデルがロードされ OCR 結果が返る |
| IT-FULL-01 | OBS 起動下でアプリを起動し記録開始→停止 | 実環境一式 | クラッシュなく記録が開始・停止できる |

> 手順書は本計画承認後に `docs/design/結合テスト手順書.md`（または報告書 §付録）として整備する。

## 4. テスト方針

- 区分Aは `pytest` で実装し、`tests/integration/` 配下に配置する（単体テストと明確に分離）。
- WebSocket の結合テストは `websockets` ライブラリのクライアント機能で実サーバへ接続する。ポートは動的に空きポートを割り当て、テスト間干渉を避ける。
- 非同期コード（`ws_server` / `recording_service`）は実スレッド・実イベントループを起動して検証する（単体テストのフェイク差し替えと異なり、本物のループを回す）。
- DB は `tmp_path` の一時 SQLite を用いる。
- 区分Aのテストは実行時間が単体テストより長くなるため、必要に応じ `pytest` マーカー（例: `@pytest.mark.integration`）で区別する。

## 6. 合否判定・完了条件

工程7は以下を**すべて**満たした時点で完了とし、`14_結合テスト報告書.md` に結果を記録、PO レビュー承認をもって工程8へ移行する。

1. 区分A（自動結合テスト）の主要シナリオが**全件パス**。
2. 区分B（実環境結合テスト）の手順書が整備され、ユーザ実施の結果が報告書に記録され、主要シナリオがパス。
3. 致命欠陥（Critical）が **0 件**。
4. `ruff check` がテストコードを含めてパス。

> 区分Aは本工程内で実施・報告できる。区分Bはユーザの実機実施を待つため、報告書は「区分A結果を先行記録し、区分B結果を追記する」運用とする。

---

## 7. 致命欠陥の定義

`11_単体テスト計画書.md` §8 と同じ等級定義（致命／重大／軽微）を用いる。発見した欠陥は `04_リスク・課題管理表.md` の課題一覧（`I-xxx`）に登録して追跡する。

## 8. 作業計画

1. `tests/integration/` を新設し、区分Aのテストを実装（フェイク OBS・空きポート割り当て等の補助を含む）。
2. `pytest` で区分A全件パス・`ruff` クリーンを確認。
3. 区分Bの実環境結合テスト手順書を整備。
4. 区分A結果を `14_結合テスト報告書.md` に記録（PO レビュー）。
5. ユーザが区分Bを実機実施 → 結果を報告書へ追記 → 工程7完了判定。

---

## 9. リスク・前提

- **R-004**：解像度・OBS レイアウトのバリエーションは、区分Bで代表的な構成を確認するに留まる。網羅検証はシステムテスト。
- **R-007**：KONAMI 側 UI 変更への耐性は、画面ごとの認識モジュール分離で担保。結合テストでは現行 UI 前提で検証する。
- 区分Bはユーザの実機環境（OBS Studio・コナステ pop'n music lively）に依存するため、本工程の完全な完了はユーザ実施の完了を要する。
- PaddleOCR モデルは初回実行時にネットワーク経由でダウンロードされる（PoC #01 で確認済み）。
